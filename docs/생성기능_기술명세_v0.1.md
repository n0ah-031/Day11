## 0. 확정된 구현 방향 (2026-07-22 의사결정)

기준 문서만으로 결정할 수 없어 확인한 4가지 사항과 결정 내용. 이후 본문은 모두 이 결정을 전제로 한다.

| # | 쟁점 | 결정 | 비고 |
|---|---|---|---|
| D1 | 양식 파일 생성 주체 | **LLM이 파일 내용을 직접 저작** — LLM이 셀 단위 명세(workbook_json: 값·병합·서식·열너비·유효성검사 전부)를 생성하고, 백엔드 materializer는 이를 xlsx로 1:1 변환만 수행(자체 레이아웃 로직 없음) | 제목행·병합셀·안내문 등 다양한 레이아웃 요구에 대응하기 위한 유연성 확보 목적. LLM이 xlsx 바이너리를 직접 출력할 수 없으므로 "셀 단위 JSON 저작 + 무해석 변환"이 기술적 구현 형태 |
| D2 | 출력 포맷 범위 | **xlsx 우선, hwpx 생성은 2차** | 본 명세는 xlsx 생성까지 확정. hwpx는 어댑터 인터페이스(출력 포맷 파라미터)만 예약 |
| D3 | 문답 흐름 제어 | **LLM 완전 주도** — 질문 순서·다음 질문·완료 판정을 모두 LLM structured output에 위임 | 완료 판정 안정화 장치(§4.3 완결성 루브릭, 최대 턴 수)를 백엔드에서 보강 |
| D4 | 첨부 컨텍스트 범위 | **xlsx + hwpx + png 전부 지원** | png는 vision 입력으로 전달. 이미지에는 정규식 마스킹 적용 불가 → §10 리스크 R3 |

> **범위 제외 (v0.3)**: 외부 기관 확정 양식을 그대로 등록·사용하는 기능은 본 명세 범위에서 제외되었다. PRD v1.5 결정 로그 #17에 따라 추후 별도 기능(F6)으로 신설하며, v0.2에서 설계했던 등록 방식(원본 무변경 등록, AI 필드 추출 + 사용자 확인, 수정 불가, 대화창 내 AI 분기)은 F6 명세 작성 시 이관한다(§11 B6).

## 2. 아키텍처

```
[화면 2·3 (React/Vite/TS, TanStack Query)]
        │ HTTPS + JWT(Supabase Auth)
        ▼
[FastAPI]
  ├─ routers/formgen.py          … §5 API
  ├─ services/intake.py          … 문답 세션 오케스트레이션 (§4)
  ├─ services/materializer.py    … workbook_json → xlsx (openpyxl) (§7)
  ├─ services/extractor.py       … 첨부(xlsx/hwpx/png) 컨텍스트 추출 (§6.5)
  ├─ security/masking.py         … 정규식 마스킹 공통 필터 (§9.2)
  └─ ai/adapter.py               … OpenAI 어댑터 (design.md §1.1 준수) (§6)
        │
        ▼
[OpenAI GPT API (structured output, vision)]      [Supabase]
                                                    ├─ Postgres: projects, intake_sessions,
                                                    │   form_templates, field_rules, audit_logs
                                                    └─ Storage: forms/(결과물), tmp/(첨부 임시)
```

- 모든 OpenAI 호출은 `ai/adapter.py` 단일 진입점 경유(벤더 교체 대비, PRD §6.5). 마스킹 필터는 어댑터 진입 **직전** 공통 미들웨어로 적용(design.md §7).
- 양식 생성(§5 generate)은 F2/F3의 `aggregation_jobs`를 쓰지 않고 `form_templates.status` 폴링으로 처리한다(단일 LLM 호출 + 변환이라 전용 작업 테이블 불요, FastAPI BackgroundTasks 사용).

### 3.3 form_templates (상세화)

| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | uuid PK | |
| project_id | uuid FK→projects | |
| intake_session_id | uuid FK→intake_sessions | 어떤 문답에서 생성됐는지 추적 |

이전 버전 파일은 Storage에 버전별 경로로 보존(이력 재사용 F4-2 대비), DB 행은 최신 버전만 갱신한다.

### 3.4 field_rules (기존, 생성 규칙 명확화)

| 컬럼 | 타입 | 설명 |
|---|---|---|
| id | uuid PK | |
| form_template_id | uuid FK | |
| field_name | text | spec_json.fields[].name과 1:1 |
| rule_type | text | `date` / `name` / `amount` / `dept_code` / `text` / `number` |
| rule_config_json | jsonb | 예: `{"format":"YYYY-MM-DD"}`, `{"strip_spaces":true}`, `{"thousands_sep":false}`, `{"required":true}` |

## 4. 상태 머신

design.md §4.1의 상태 머신을 세션·템플릿 2계층으로 구체화한다.

```
[intake_sessions.status]                    [form_templates.status]
 active ──(LLM 완료 판정)──▶ spec_complete ──(generate 요청)──▶ generating ──▶ done
   ▲                              │                                  │           │
   └──(사용자 추가 답변으로 사양 변경)┘                                  └─▶ failed   │
                                                                        (revise)◀──┘
                                                              revise: done ─▶ generating ─▶ done(v+1)
```

### 4.1 전이 규칙

| 전이 | 트리거 | 비고 |
|---|---|---|
| active → spec_complete | 어댑터 응답 `spec_complete=true` | 프론트 생성 버튼 활성화는 **서버가 내려준 이 플래그로만** 결정(클라이언트 자체 판정 금지) |
| spec_complete → active | 사용자가 추가 메시지 전송(사양 변경) | 완료 후에도 문답 계속 가능. 재완료 판정 필요 |
| spec_complete → (template) generating | `POST .../generate` | 세션 status는 `closed`로, 첨부 임시파일 삭제(§6.5) |
| generating → done | materializer 성공, file_url 확정 | |
| generating → failed | LLM 저작 실패(재시도 소진) 또는 변환 실패 | error_message 기록, 프론트에 재시도 버튼 노출 |
| done → generating (revise) | `POST /form-templates/{id}/revise` | version+1. 실패 시 기존 버전 유지한 채 failed 표시가 아니라 **직전 done 상태로 롤백**하고 오류 토스트만 표시 |

### 4.2 LLM 완전 주도 문답 (D3)

- 매 턴 어댑터에 전달: 전체 대화 이력(마스킹 적용) + 현재 spec_json + 첨부 추출 컨텍스트.
- 어댑터 응답(structured output 강제): `{ reply, spec_json(전체 갱신본), spec_complete, coverage }`.
- 질문 내용·순서·세분화 수준은 전부 LLM 판단. 시스템 프롬프트로 "PRD 인터뷰 방식: 대상 부서 → 포함 항목 → 항목별 입력 형식(날짜 표기·숫자 단위 등) → 필수/선택 구분을 반드시 확인하고, 한 번에 한 주제씩 질문"을 지시한다(PRD F1-2의 필수 확인 항목을 프롬프트 계약으로 보장).

### 4.3 완료 판정 안정화 (백엔드 가드)

LLM 완전 주도의 흔들림을 막기 위한 서버측 최소 가드 2개:

1. **완결성 루브릭 검증**: `spec_complete=true` 응답이라도 백엔드가 spec_json을 검사해 다음이 비면 강제로 `false` 처리하고 어댑터에 부족 항목을 알려 재질문시킨다 — `form_title` 존재, `fields` 1개 이상, 모든 field에 `type`·`required` 존재, date/amount 타입 field에 `format` 존재.
2. **최대 턴 수**: `turn_count > 30`이면 어댑터에 "지금까지 정보로 사양을 마감하라"는 마감 지시를 보내고, 그래도 루브릭 미달이면 부족 항목을 사용자에게 직접 목록으로 제시한다.

## 5. API 명세

모든 엔드포인트: JWT 필수, `projects.owner_id` 스코핑, 실패 응답은 `{error: {code, message}}` 공통 포맷.

### 5.1 동의 (선행 조건)

| 메서드/경로 | 요청 | 응답 |
|---|---|---|
| `POST /auth/ai-consent` | (없음) | `204` — profiles.ai_consent_at 기록 |
| `GET /auth/me` (기존 확장) | | `{ ..., ai_consented: bool }` |

`ai_consented=false`인 사용자의 F1 API 호출은 전부 `403 AI_CONSENT_REQUIRED` → 프론트가 동의 모달 표시(§9.1).

### 5.2 문답 (INTAKE)

**`POST /projects` (기존)** — 화면 2에서 첫 프롬프트 전송 시 프론트가 먼저 호출. `{name?: string}` → `{project_id}`. name 생략 시 "새 양식"으로 두고 첫 턴 후 LLM 요약으로 갱신.

**`POST /projects/{id}/form-intake/messages`**

```jsonc
// 요청
{ "message": "부서별 예산 집행 현황 양식 만들어줘", "attachment_ids": ["..."] }  // attachment_ids 선택

// 응답 200
{
  "session_id": "uuid",
  "reply": "이 양식을 받을 대상 부서를 알려주세요. ...",
  "spec_json": { /* §6.2 전체 갱신본 */ },
  "spec_complete": false,
  "coverage": { "confirmed_topics": ["양식명","대상 부서"], "remaining_topics": ["포함 항목","입력 형식","필수/선택"] }
}
```

- 활성 세션이 없으면 생성, 있으면 이어간다(멱등적 세션 관리).
- `coverage`는 LLM이 자체 보고하는 참고값 — 화면 3 진행률 UI의 데이터 소스(§8.2). 신뢰도 한계는 §11 보완 B3.
- 처리 시간 목표 8초 이내. 스트리밍은 MVP 미적용(§12 판단 J2).

**`POST /projects/{id}/form-intake/attachments`** (multipart)

```jsonc
// 응답 200
{ "attachment_id": "uuid", "original_name": "기존양식.hwpx", "kind": "hwpx",
  "extracted_summary": "표 1개(열: 부서, 항목, 금액...), 텍스트 1,240자 추출" }
```

- 허용 확장자 `xlsx|hwpx|png`, 파일당 50MB(PRD §6.1 업로드 제한 준용 — §12 판단 J3), 세션당 최대 5개.
- 바이너리는 Storage `tmp/{session_id}/`에 저장, **세션 close 시(생성 실행 또는 7일 경과) 삭제**. DB에는 추출 결과만 잔존(design.md "별도 저장하지 않음" 원칙과 정합).

**`GET /projects/{id}/form-intake/session`** — 화면 3 재진입/새로고침 복원용. `{session_id, messages, spec_json, spec_complete, coverage}`.

### 5.3 생성 (GENERATING)

**`POST /projects/{id}/form-templates/generate`**

```jsonc
// 요청
{ "session_id": "uuid", "output_format": "xlsx" }   // format은 현재 xlsx만 허용
// 응답 202
{ "template_id": "uuid", "status": "generating" }
```

- 전제: 세션 `spec_complete`. 아니면 `409 SPEC_NOT_COMPLETE`.
- BackgroundTasks로 §6.3 저작 + §7 변환 실행.

**`GET /form-templates/{id}`** — 폴링(2초 간격 권장).

```jsonc
{ "template_id": "uuid", "status": "done", "version": 1,
  "spec_json": {...}, "workbook_json": {...},        // 미리보기 렌더용
  "download_url": "https://...signed...", "error_message": null }
```

`download_url`은 Supabase Storage signed URL(만료 10분, 요청 시마다 재발급).

### 5.4 수정 (F1-7)

**`POST /form-templates/{id}/revise`**

```jsonc
// 요청
{ "instruction": "비고 컬럼을 추가하고 금액은 천 단위 콤마로 바꿔줘" }
// 응답 202
{ "template_id": "uuid", "status": "generating", "version": 2 }
```

- 어댑터에 현재 workbook_json + spec_json + 수정 지시를 전달, 전체 갱신본을 받아 재변환.
- spec_json도 함께 갱신본을 받아 field_rules 재파생(§3.4).
- 이후 `GET /form-templates/{id}` 폴링 동일.

### 6.1 인터페이스

```python
class FormGenAdapter(Protocol):
    def intake_turn(self, history: list[Msg], spec: dict,
                    attachments: list[ExtractedDoc]) -> IntakeTurnResult:
        """reply, spec_json 갱신본, spec_complete, coverage 반환"""

    def author_workbook(self, spec: dict,
                        attachments: list[ExtractedDoc]) -> dict:
        """workbook_json(§6.3) 저작. D1: 레이아웃·서식 결정권 전부 LLM"""

    def revise_workbook(self, workbook: dict, spec: dict,
                        instruction: str) -> ReviseResult:
        """workbook_json·spec_json 전체 갱신본 반환"""
```

- 구현체 `OpenAIFormGenAdapter` 1개. 모델명·temperature는 환경변수(`FORMGEN_MODEL` 등)로 주입.
- 세 메서드 모두 JSON Schema 기반 **structured output을 강제**한다. 스키마 불일치 응답은 오류 메시지를 포함해 최대 2회 재요청, 소진 시 예외 → 상태 failed.
- zero data retention: OpenAI API 기본 정책(입력 미학습) 전제 + 조직 설정으로 데이터 보존 옵트아웃 적용(PRD §6.2). 어댑터에서 별도 헤더/설정 확인 로직은 두지 않되 배포 체크리스트에 포함.

```jsonc
{
  "form_title": "부서별 예산 집행 현황",
  "target_depts": ["전체 부서"],                    // 또는 명시 목록
  "fields": [
    { "name": "부서명",   "type": "dept_code", "required": true,  "format": null,
      "notes": "사전 정의된 코드표 기준" },

    { "name": "집행금액", "type": "amount",    "required": true,  "format": "숫자만, 구분기호 없음", "notes": null },
    { "name": "비고",     "type": "text",      "required": false, "format": null, "notes": null }

  "layout_hints": "월별 시트 분리, 상단에 제목 행",   // 자유 텍스트 (참고 첨부 반영 요구 포함)
```

`type` enum: `text | number | amount | date | name | dept_code` — field_rules.rule_type과 정렬(§3.4).

### 6.3 workbook_json 스키마 (파일 내용 — LLM 저작물, D1의 핵심)

```jsonc
{
  "sheets": [
    {
      "name": "양식",                                  // 31자 제한, 중복 금지 (백엔드 검증)
      "columns": [ { "index": 1, "width": 14 } ],
      "merges": [ "A1:F1" ],
      "freeze_panes": "A4",
      "cells": [
        { "ref": "A1", "value": "2026년 부서별 예산 집행 현황", "style": "title" },
        { "ref": "A3", "value": "부서명", "style": "header" },
        { "ref": "B3", "value": "집행일자", "style": "header" }
      ],
      "styles": {                                      // LLM이 정의하는 named style
        "title":  { "bold": true, "size": 14, "align": "center" },
        "header": { "bold": true, "bg": "EEF2FF", "border": "thin", "align": "center" }
      },
      "validations": [
        { "range": "B4:B200", "type": "date",
          "prompt": "YYYY-MM-DD 형식으로 입력", "error": "날짜 형식이 올바르지 않습니다" },
        { "range": "A4:A200", "type": "list", "source": ["기획팀","인사팀","총무팀"] }
      ]
    }
  ]
}
```

**백엔드 검증 규칙(materializer 진입 전, LLM 산출물 신뢰 금지):**

| 항목 | 제한 |
|---|---|
| 시트 수 | ≤ 10 |
| 셀 수 합계 | ≤ 10,000 |
| 시트명 | ≤ 31자, `\/?*[]:` 금지, 중복 금지 |
| ref/range | A1 표기법 정규식 검증 |
| style 속성 | 화이트리스트(bold, italic, size, bg, color, border, align, number_format)만 통과 |
| value | 문자열/숫자만. `=`로 시작하는 수식은 허용 함수 화이트리스트(SUM, SUMIF, SUMIFS, COUNT, IF, TODAY) 검사, 외부 참조(`[`, `!`가 다른 문서 가리킴) 금지 |

검증 실패 시 실패 항목을 어댑터에 피드백하여 재저작(최대 2회) 후 소진 시 failed.

### 6.4 GPT API 연결 설정 및 API 키 관리

- **사용 라이브러리**: OpenAI 공식 Python SDK(`openai>=1.x`). `AsyncOpenAI` 클라이언트는 `ai/adapter.py` 모듈 로드 시 1회만 초기화해 재사용한다(요청마다 재생성 금지).

```python
# ai/adapter.py
from openai import AsyncOpenAI
from core.config import settings

client = AsyncOpenAI(
    api_key=settings.openai_api_key,
    organization=settings.openai_org_id,   # None이면 SDK가 무시
    timeout=30.0,                          # §9.4 어댑터 호출 타임아웃과 정합
)
```

- **환경변수 목록**

| 변수명 | 필수 | 설명 |
|---|---|---|
| `OPENAI_API_KEY` | Y | OpenAI 플랫폼에서 발급한 시크릿 키. 코드·저장소에 하드코딩 절대 금지 |
| `FORMGEN_MODEL` | Y | `intake_turn`/`author_workbook`/`revise_workbook`(§6.1)에 공통 사용할 모델명 |
| `FORMGEN_VISION_MODEL` | N | png 첨부(vision, §6.5) 처리에 별도 모델을 쓸 경우만 지정. 미지정 시 `FORMGEN_MODEL` 사용 |
| `OPENAI_ORG_ID` | N | zero data retention(PRD §6.2)이 조직 단위로 설정된 경우에만 사용 |

- **설정 로더 (`core/config.py`)**: `pydantic-settings`로 `.env`를 로드해 타입 검증. 필수 변수가 비어 있으면 **앱 부팅 시점에 즉시 실패**(fail-fast)하도록 하여, 런타임 중 키 누락으로 인한 어댑터 재시도·타임아웃 소진(§9.4)을 방지한다.

```python
from pydantic_settings import BaseSettings

class Settings(BaseSettings):
    openai_api_key: str
    formgen_model: str = "gpt-5"
    formgen_vision_model: str | None = None
    openai_org_id: str | None = None

    class Config:
        env_file = ".env"

settings = Settings()
```

- **`.env` 저장·관리 원칙**
  1. 로컬 개발: 저장소 루트의 `.env` 파일에 키를 저장. 저장소 `.gitignore`에 `.env`가 등록되어 있음을 유지·확인(커밋 금지).
  2. 저장소에는 `.env.example`만 커밋해 필요한 변수명과 형식을 안내하고, 실제 값은 비워 둔다.
  3. FastAPI 앱은 `core/config.py`의 `settings`만 참조하며, `ai/adapter.py`를 포함한 다른 모듈은 `os.environ`을 직접 읽지 않고 항상 `settings`를 주입받아 사용한다(키 접근 경로 단일화).
  4. 운영 배포 환경에서는 `.env` 파일을 사용하지 않고, 배포 플랫폼이 제공하는 환경변수/시크릿 매니저에 동일한 키를 등록한다. 로컬 `.env`와 운영 시크릿 값은 서로 다르며, 운영 키는 코드 저장소 밖의 별도 채널로만 공유한다.
  5. 키 노출(오커밋, 로그 유출 등) 발생 시 즉시 OpenAI 플랫폼에서 해당 키를 폐기(revoke)하고 재발급하며, `audit_logs`(§9.3)에 사건을 기록한다.

### 6.5 첨부 추출 파이프라인 (`services/extractor.py`, D4)

| kind | 추출 방법 | 어댑터 전달 형태 |
|---|---|---|
| xlsx | openpyxl로 시트별 헤더·데이터 영역·병합·서식 요약을 구조화 텍스트로 직렬화 (상위 100행) | 텍스트 (마스킹 적용) |

| png | 추출 없이 원본 이미지 전달 | vision 이미지 입력 (**마스킹 불가** — §10 R3) |

## 7. Materializer (`services/materializer.py`)

- 입력: 검증 통과한 workbook_json. 출력: xlsx 바이트 → Storage 업로드 → file_url.
- openpyxl로 **무해석 1:1 변환**: 자체 레이아웃·기본값 주입 로직을 두지 않는다(D1 원칙 — 레이아웃 품질 책임은 LLM+검증 루프에 있음). 유일한 예외는 안전 기본값: 정의 안 된 style 참조는 무서식 처리, width 미지정 열은 auto 12.
- `validations`는 openpyxl DataValidation으로 구현(date → 커스텀 수식 기반, list → 드롭다운).
- 단위 테스트: workbook_json 픽스처 → 생성된 xlsx를 재로드하여 셀 값·병합·유효성검사 존재를 assert.

| 프로토타입 요소 | 구현 치환 |

| `addSpecCard(label, value)` 누적 방식 | 매 응답의 `spec_json` 전체로 우측 카드 **전량 재렌더**(필드 수정·삭제 반영 위해 누적 아님). 카드 구성: 양식명 / 대상 부서 / 필드별 카드(이름·타입·형식·필수 배지) |

## 9. 보안·운영

### 9.1 AI 전송 동의 (PRD §6.2)

- 최초 F1 진입 시 1회 모달: "입력 내용과 첨부 파일이 OpenAI API로 전송됩니다(학습 미사용). **이미지 첨부는 자동 마스킹이 적용되지 않으니 개인정보가 포함된 이미지는 첨부하지 마세요.**" — 동의 시 `POST /auth/ai-consent`.
- 동의 전 F1 API는 403(§5.1).

### 9.2 개인정보 마스킹 (`security/masking.py`)

- 적용 지점: 어댑터로 나가는 **모든 텍스트**(사용자 메시지, 대화 이력, 첨부 추출 텍스트). DB 저장본에는 미적용(내부 보관, RLS로 격리).
- 자동 치환(정규식): 주민등록번호 `\d{6}-?[1-4]\d{6}` → `[주민번호]`, 휴대전화 `01[016789]-?\d{3,4}-?\d{4}` → `[전화번호]`, 이메일 → `[이메일]`.
- 성명 열: 첨부 xlsx 추출 시 헤더가 성명/이름 계열이면 업로드 응답에 `has_name_column: true`를 반환하고, 프론트가 "성명 데이터를 마스킹할까요?" 토글 제시 → 세션 첨부 메타에 저장, 추출 텍스트 재생성.
- png은 적용 불가(동의 문구로 고지, §10 R3).

### 9.3 감사 로그

`audit_logs`에 기록: `form_intake_message`, `form_intake_attachment`, `form_generate`, `form_revise`, `form_download` (actor_id, target=template/session id). Admin API 비용 대시보드용으로 어댑터가 호출당 토큰 사용량을 `audit_logs.action='ai_usage'` 행으로 남긴다(비용 산출은 Admin 기능 범위).

### 9.4 AI 장애 시 기능 저하 모드 (PRD §6.4)

- 어댑터 호출 타임아웃 30초, 재시도 1회(지수 백오프). 최종 실패 시:
  - intake: 채팅에 시스템 버블 "AI 서비스가 일시적으로 불안정합니다. 잠시 후 다시 시도해주세요" + 입력 유지(대화 이력 보존으로 재시도 시 이어짐).
  - generate/revise: status=failed + error_message, 프론트 재시도 버튼.
- OpenAI 5xx가 5분 내 N회 누적 시 화면 2 '생성' 탭 상단에 저하 모드 배너(서버 헬스 플래그).
