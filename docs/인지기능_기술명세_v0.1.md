## 0. 확정된 구현 방향

이관 결정 4건(생성기능_기술명세_v0.1.md v0.2에서 설계, §11 B6에서 이관 지시)과 본 명세 작성을 위해 신규로 확인한 4건.

| # | 쟁점 | 결정 | 출처 |
|---|---|---|---|
| E1 | 원본 파일 처리 방식 | **무변경 등록** — 업로드된 파일을 LLM이 재저작하지 않고 바이트 그대로 저장. F1의 materializer(워크북 재변환)를 거치지 않는다 | 이관(v0.2) |
| E2 | 정합성 기준 확보 방식 | AI가 구조를 인지해 필드·작성기준을 추출하고, **사용자가 확인·보정**하는 절차를 거쳐 확정 | 이관(v0.2) |
| E3 | 등록 후 수정 정책 | 등록된 **원본 파일(레이아웃·서식)은 수정 불가**(재저작 없음, F1-7 같은 revise 없음). 단 추출된 작성기준(field_rules)은 등록 후에도 수정 가능 | 이관(v0.2) |
| E4 | 진입 경로 | 별도 신규 화면이 아니라 **화면 3(F1 문답 화면)의 대화창에서 AI가 사용자 의도를 판정해 자동 분기** | 이관(v0.2) |
| E5 | 업로드 대상 파일의 상태 | **빈 양식과 예시(샘플 데이터 포함) 파일 모두 지원**. 값이 있으면 형식(rule_config_json) 추론에 활용하고, 없으면 헤더·기존 서식·기존 유효성 검사만으로 추론 | 신규(본 세션) |
| E6 | 인지 결과의 저장·재사용 범위 | F1과 동일하게 **프로젝트 단위 `form_templates`/`field_rules`로 저장**해 이후 여러 차례의 F2 취합 작업에서 재사용 | 신규(본 세션) |
| E7 | 확인 질문 방식 | **자동 추론을 우선**하고, 확신도가 낮거나 필수 정보가 빠진 필드만 **최소한의 확인 질문**으로 보완(F1-2 같은 전면 문답이 아님) | 신규(본 세션) |
| E8 | 실행 시점 | F2 이전 별도의 "양식 등록" 사전 화면이 아니라, **F1 생성 화면(화면 3)에 통합된 내장 스텝**으로 실행(E4의 구체화) | 신규(본 세션) |

> E1·E3에 따라 F6에는 F1의 `materializer`·`revise` 대응 기능이 존재하지 않는다. 대신 원본 파일을 그대로 보존하는 **등록(register)** 절차와, 작성기준만 별도로 고치는 **field_rules 보정** 절차로 나뉜다(§7, §5).

## 1. 기능 범위 요약 (제안 F6 요구사항 — PRD 미반영, 본 명세에서 초안 제시)

PRD v1.5는 F6을 "추후 신설(결정 로그 #17)"로만 예약해두었다. 아래 ID는 본 명세가 제안하는 요구사항 목록이며, PRD §4에 F6 절 신설이 필요하다(§11 C1).

| ID | 요구사항 | 본 명세 반영 위치 |
|---|---|---|
| F6-1 | 화면 3 대화창에서 사용자의 발화·첨부로부터 "기존에 정해진 외부 양식을 그대로 쓰겠다"는 의도를 AI가 판정해 인지·등록 플로우로 자동 분기. 모호하면 재확인 | §4.2 |
| F6-2 | 외부 양식 파일 업로드 — xlsx(완전 지원, 빈 양식·예시 데이터 모두 허용), hwpx(참고 첨부로만 허용, 등록 대상 아님), png(미지원) | §6.5, §9 |
| F6-3 | 구조 인지 — 업로드 xlsx의 시트·헤더·데이터 영역·기존 서식(number_format)·기존 유효성 검사(목록 등)를 구조적으로 추출하고, 이를 근거로 필드별 타입·형식·필수여부를 AI가 추론 | §6.2, §6.3 |
| F6-4 | 확인·보정 — 추론 결과를 필드별로 제시하고, 확신도가 낮은 항목만 최소 확인 질문으로 보완. 등록 확정 전 사용자가 임의 필드를 직접 수정 가능 | §4.3, §8 |
| F6-5 | 등록 확정 — 원본 파일을 무변경 저장(§7 Registrar)하고, 확정된 필드 정보를 작성기준(field_rules)으로 저장 | §5, §7 |
| F6-6 | 취합 연동 — 등록된 양식·작성기준은 F2-5 전처리에서 F1 생성 양식과 동일한 방식으로 자동 참조 | §3, §11 C3 |
| F6-7 | 등록 후 보정 — 원본 파일 자체는 수정 불가하나 field_rules는 등록 후에도 재확인·수정 가능. 원본 양식이 바뀌면 재인지(새 버전 등록)로 처리 | §5, §12 J2 |

비기능 요구 반영: AI 전송 고지·동의(§9.1, F1과 공유 + 예시데이터 특이사항), 개인정보 마스킹(§9.2, 샘플 데이터 리스크 강화), 원본 파일 무결성 검증(§9.3), 감사 로그(§9.4).

## 2. 아키텍처

```
[화면 3 (F1과 공유, React/Vite/TS)]
        │ HTTPS + JWT(Supabase Auth)
        ▼
[FastAPI]
  ├─ routers/formgen.py            … §5 API (F1 라우터 확장)
  ├─ services/intake.py            … 의도 판정 + generate/recognize 분기 오케스트레이션 (§4)
  ├─ services/form_structure.py    … xlsx 구조(헤더·서식·유효성검사·샘플행) 추출 (§6.2, 신규)
  ├─ services/form_registrar.py    … 원본 파일 무변경 등록 + field_rules 파생 (§7, 신규)
  ├─ services/extractor.py         … hwpx 참고 첨부 텍스트 추출 (F1 §6.5 재사용)
  ├─ security/masking.py           … 정규식 마스킹 공통 필터 (F1 §9.2 재사용)
  └─ ai/adapter.py                 … OpenAI 어댑터 (classify_intent·recognize_turn 추가, §6)
        │
        ▼
[OpenAI GPT API (structured output)]              [Supabase]
                                                    ├─ Postgres: intake_sessions(확장),
                                                    │   form_templates(확장), field_rules, audit_logs
                                                    └─ Storage: forms/(등록 원본), tmp/(첨부 임시)
```

- `form_registrar.py`는 F1 `materializer.py`와 달리 **openpyxl로 파일을 열고 다시 쓰지 않는다** — 원본 바이트를 그대로 이동시켜 서식·매크로·조건부서식 손실을 방지한다(E1).
- 등록(register)은 F1의 생성(generate)과 달리 **동기 처리**다(§5, §12 J1) — LLM이 파일을 저작하지 않고 구조 추출 + field_rules upsert만 수행하므로 `aggregation_jobs`나 폴링이 불필요하다.

## 3. 데이터 모델

design.md §3 / 생성기능_기술명세_v0.1.md §3의 `intake_sessions`, `form_templates`, `field_rules`를 확장한다. 신규 테이블은 없다.

### 3.1 intake_sessions 확장

| 신규 컬럼 | 타입 | 설명 |
|---|---|---|
| `mode` | text, nullable | `generate` / `recognize` / `null`(첫 턴 이전, 의도 미판정) |
| `primary_attachment_id` | uuid, nullable | `recognize` 모드에서 "등록 대상 원본 파일"로 지정된 첨부 id. 여러 xlsx 첨부 시 사용자가 명시적으로 지정 |
| `structural_json` | jsonb, nullable | `recognize` 모드의 구조 추출 결과(§6.2 스키마). LLM 입력이자 화면 3 미리보기·검수 표의 공용 소스 |

`spec_json`·`spec_complete`·`turn_count` 컬럼은 F1과 동일하게 재사용하되, `recognize` 모드에서는 각 field에 `confidence` 속성이 추가된다(§6.3). `spec_complete`의 완결성 판정 루브릭은 모드별로 다르다(F1 §4.3 vs 본 문서 §4.3).

### 3.2 form_templates 확장

| 신규/변경 컬럼 | 타입 | 설명 |
|---|---|---|
| `source` | text, **신규 필수** | `ai_generated` / `recognized_external`. 기존 F1 행은 `ai_generated`로 backfill |
| `workbook_json` | jsonb, **nullable로 변경** | `recognized_external`인 경우 NULL(원본 파일 자체가 산출물이므로 셀 단위 재현 명세 불필요) |
| `structural_json` | jsonb, nullable, 신규 | `recognized_external`인 경우 등록 시점의 구조 추출 스냅샷 보존(재인지·이력 대비). `ai_generated`는 NULL |
| `file_url` | text | `recognized_external`인 경우 업로드 원본을 무변형 복사한 경로 `forms/{project_id}/{template_id}_v{version}.xlsx` |
| `output_format` | text | `recognized_external`인 경우 업로드 원본 확장자(`xlsx`만 정식 지원, §6.5) |

`version`·`error_message`·`created_at`/`updated_at`은 F1과 동일 의미로 재사용. `recognized_external`은 `error_message`가 사실상 쓰이지 않는다(§5 register가 동기·즉시 실패 응답이라 상태로 남기지 않음).

### 3.3 field_rules — 변경 없음

F1 §3.4 스키마·파생 로직을 그대로 재사용한다. `form_template_id`가 `recognized_external` 소스를 가리키는 경우에도 동일하게 동작한다(F6-6의 근거).

## 4. 상태 머신

### 4.1 전체 흐름

```
[intake_sessions.mode]
 (null) ──(첫 턴 intent 판정: generate)──▶ 이후 생성기능_기술명세_v0.1.md §4 그대로
 (null) ──(첫 턴 intent 판정: ambiguous)──▶ (null 유지, 재확인 질문 1회, §4.2)
 (null) ──(첫 턴 intent 판정: recognize)──▶ recognize
                                              │
                    ┌─────────────────────────┘
                    ▼
      active ──(recognize_complete 루브릭 통과)──▶ spec_complete ──(register 요청, 동기)──▶ form_templates.done
        ▲                    │
        └──(사용자 보정 답변)──┘

form_templates.done ──(PATCH field-rules)──▶ done (파일·version 불변, field_rules만 갱신)
form_templates.done ──(POST re-register, 새 파일)──▶ done(version+1, 구조·field_rules 전체 재추출)
```

### 4.2 의도 판정 (E4/E8 구체화)

- 첫 턴 응답에 `intent` 필드 추가: `generate` / `recognize` / `ambiguous`.
- 판정 근거(시스템 프롬프트 계약): "이미 정해진/확정된 양식", "이 양식 그대로", "외부에서 받은 양식" 등의 표현과 함께 xlsx 첨부가 있으면 `recognize`. 첨부 없이 순수 신규 요청이면 `generate`. 첨부는 있으나 발화가 모호하면 `ambiguous`.
- `ambiguous`인 동안 `mode`는 `null`로 유지되고, 응답 `reply`에 재확인 질문이 포함된다. 화면 3은 이때 리플라이 버블 아래에 "새 양식 만들기" / "첨부한 파일 그대로 등록" 버튼 2개를 인라인 노출해 **사용자가 직접 분기를 확정**할 수 있게 한다(무한 루프 방지 안전장치, §8).

### 4.3 recognize 완결성 루브릭 (백엔드 가드, F1 §4.3 대응)

`recognize_complete=true` 응답이라도 백엔드가 아래를 검사해 미달 시 강제로 `false` 처리하고 부족 항목을 재질문시킨다.

1. `structural_json`에서 식별된 모든 헤더가 `spec_json.fields`에 **빠짐없이 1:1 매핑**되어 있을 것
2. 모든 field에 `type`·`required` 존재
3. `date`/`amount` 타입 field는 `format` 존재(구조 추출의 `existing_number_format` 힌트로 해결 못하면 확인 질문 필수 유발)
4. 최대 턴 수: `turn_count > 10`(F1의 30턴보다 짧다 — 구조가 이미 주어져 확인 질문 개수가 적은 것이 정상, §12 J3). 초과 시 부족 항목을 목록으로 직접 제시

## 5. API 명세

F1의 `POST /projects`, `POST /projects/{id}/form-intake/messages`, `POST /projects/{id}/form-intake/attachments`, `GET /projects/{id}/form-intake/session`을 그대로 재사용하되 아래와 같이 확장·추가한다.

### 5.1 재사용 엔드포인트 확장

- **`POST /projects/{id}/form-intake/messages`** 응답에 `mode`(첫 턴 이후 고정), 첫 턴에 한해 `intent` 추가. `recognize` 모드에서는 `spec_json.fields[].confidence`, `structural_json`이 함께 내려간다.
- **`GET /projects/{id}/form-intake/session`** 응답에 `mode`, `structural_json`, `primary_attachment_id` 포함하도록 확장(새로고침 복원용).

### 5.2 신규 — 등록 대상 지정 (첨부 2개 이상일 때만)

**`PATCH /projects/{id}/form-intake/attachments/{attachment_id}/mark-primary`**

```jsonc
// 응답 204
```
- xlsx 첨부가 1개뿐이면 자동으로 `primary_attachment_id`가 지정되어 이 API 호출이 불필요하다.

### 5.3 신규 — 등록 확정 (동기)

**`POST /projects/{id}/form-templates/register`**

```jsonc
// 요청
{ "session_id": "uuid" }

// 응답 200 (동기, 202/폴링 없음)
{
  "template_id": "uuid", "status": "done", "version": 1,
  "source": "recognized_external",
  "spec_json": { /* confidence 포함 §6.3 */ },
  "download_url": "https://...signed...(10분)"
}
```

- 전제: `session.mode='recognize'` ∧ `spec_complete=true`. 아니면 F1과 동일하게 `409 SPEC_NOT_COMPLETE`.
- 처리(§7 Registrar): 원본 파일 무변형 이동, 확장자·매직바이트 재검증, `field_rules` upsert, 세션 `closed` 전환 + 첨부 임시파일 정리.
- 처리 실패(파일 손상·검증 실패) 시 `400 REGISTER_FAILED`로 즉시 응답(비동기 `failed` 상태 없음, §12 J1).

### 5.4 신규 — 작성기준 보정 (F6-7)

**`PATCH /form-templates/{id}/field-rules`**

```jsonc
// 요청
{ "field_rules": [ { "id": "uuid?", "field_name": "집행일자", "rule_type": "date", "rule_config_json": {"format":"YYYY-MM-DD","required":true} } ] }
// 응답 200
{ "field_rules": [ /* 갱신된 전체 목록 */ ] }
```

- `source` 무관하게 사용 가능하나 실질적으로 `recognized_external`에서 주로 쓰인다(E3 — 원본 파일은 불변, 작성기준만 보정).
- 원본 파일(`file_url`)·`version`에는 영향 없음.

### 5.5 신규 — 재인지(새 버전 등록, F6-7)

**`POST /form-templates/{id}/re-register`**

- 화면 3을 새 세션으로 재진입해 `recognize` 모드를 다시 수행하고, 완료 시 이 엔드포인트로 `version+1`을 생성하는 흐름을 권장한다(전용 화면 없음, §12 J2). 요청/응답 구조는 §5.3과 동일하되 `version`이 증가한다.

## 6. AI 어댑터 설계 확장

### 6.1 인터페이스 확장

```python
class FormGenAdapter(Protocol):
    # ...F1 §6.1 기존 3개 메서드(intake_turn, author_workbook, revise_workbook)...

    def classify_intent(self, first_message: str, has_xlsx_attachment: bool) -> IntentResult:
        """intent: generate | recognize | ambiguous 판정 (첫 턴 전용)"""

    def recognize_turn(self, history: list[Msg], spec: dict,
                        structural: dict) -> RecognizeTurnResult:
        """reply, spec_json(필드별 confidence 포함) 갱신본, recognize_complete 반환"""
```

- `classify_intent`와 최초 `recognize_turn`은 구현 시 1회 호출로 합칠 수 있다(인터페이스만 규정, 최적화는 자유).
- structured output 강제, 스키마 불일치 시 최대 2회 재요청 — F1 §6.1 정책 재사용.

### 6.2 구조 추출 스키마 (`services/form_structure.py`, 신규)

xlsx 전용. F1 §6.5의 텍스트 요약과 달리 **구조화된 JSON**을 직접 생성한다(openpyxl).

```jsonc
{
  "sheets": [
    {
      "name": "예산집행현황",
      "header_row": 3,
      "headers": ["부서명", "집행일자", "집행금액", "비고"],
      "columns": [
        { "letter": "A", "existing_number_format": "General", "existing_validation": { "type": "list", "source": ["기획팀","인사팀","총무팀"] } },
        { "letter": "B", "existing_number_format": "yyyy-mm-dd", "existing_validation": null },
        { "letter": "C", "existing_number_format": "#,##0", "existing_validation": null },
        { "letter": "D", "existing_number_format": "General", "existing_validation": null }
      ],
      "sample_rows": [ { "부서명": "기획팀", "집행일자": "2026-03-02", "집행금액": 1200000, "비고": "" } ]
    }
  ]
}
```

- `sample_rows`는 **예시(샘플 데이터 포함) 파일을 업로드한 경우에만** 채워지고, 빈 양식이면 빈 배열(E5).
- 시트 수·셀 수 상한은 F1 §6.3 검증 규칙을 재사용(시트 ≤10, 셀 합계 ≤10,000) — 초과 시 등록 거부.
- `sample_rows`의 값은 **마스킹 파이프라인(§9.2)을 거친 후에만** 어댑터로 전달한다. `structural_json`의 DB 저장본은 마스킹 이전 원문(내부 보관, RLS 격리 — F1 §9.2 정책과 동일).
- 이 JSON은 어댑터 입력이자, 화면 3 우측 패널의 "원본 미리보기" 및 field_rules 검수 표 렌더링의 공용 소스로 쓰인다(F1의 `workbook_json` 미리보기에 대응).

### 6.3 spec_json 확장 (confidence)

F1 §6.2 스키마에 필드별 `confidence`만 추가한다.

```jsonc
{ "name": "집행일자", "type": "date", "required": true, "format": "YYYY-MM-DD", "notes": null, "confidence": "high" }
```

- `confidence: "low"`인 필드가 §4.3 루브릭 조건과 겹치면(예: date인데 format을 구조에서 못 찾음) 반드시 확인 질문 대상이 된다. 구조로는 이미 명확하지만 업무적 의미가 모호한 경우(예: 필수 여부가 셀만으로는 판단 불가)에도 `low`로 표시해 확인을 유도한다.

### 6.4 hwpx 참고 첨부

- F6-2에 따라 hwpx는 등록 대상이 될 수 없다. F1 §6.5 `extractor.py`의 텍스트 추출을 그대로 재사용해 **참고 컨텍스트로만** 어댑터에 전달한다(필드 추출 근거로 사용하되 field_rules 파생·등록 대상 지정 불가).
- png는 F6에서 아예 미지원(첨부 자체를 거부하고 인라인 안내, §8).

## 7. 등록 처리 (Registrar, `services/form_registrar.py`)

F1의 `materializer.py`에 대응하지만 훨씬 단순하다 — **콘텐츠를 만들지 않고 보존만 한다**.

- **입력**: `primary_attachment_id`가 가리키는 tmp 파일, 확정된 `spec_json`.
- **처리**:
  1. 확장자·매직바이트 검증(xlsx는 zip 시그니처 `PK\x03\x04` 확인 — 확장자 위변조 방지)
  2. **매크로 포함 확장자(.xlsm 등) 거부**(§9.3, §12 J5) — 보안 리스크로 인해 xlsx만 허용
  3. Storage `tmp/{session_id}/` → `forms/{project_id}/{template_id}_v{version}.xlsx`로 **이동**(openpyxl로 열고 재저장하지 않음 — 재저장 시 조건부서식·매크로·원본 서식 손실 위험 방지)
  4. `spec_json.fields` → `field_rules` upsert(F1 §3.4 로직 그대로 재사용, LLM 재호출 없음)
- **출력**: `file_url`, `field_rules` 목록.
- **단위 테스트**: 병합 셀·조건부서식이 포함된 실제 기관 양식 샘플을 등록한 후 **원본 바이트 해시가 등록 전후로 동일**한지 assert(무변형 등록 검증, E1의 핵심 보증).

## 8. 프론트엔드 명세 (화면 3 확장)

화면 3은 F1과 완전히 공유하는 단일 화면이며, `mode`에 따라 우측 패널과 하단 액션 영역만 달라진다.

| 요소 | 생성 모드(F1, 기존) | 인지 모드(F6, 신규) |
|---|---|---|
| 우측 패널 제목 | "지금까지 확인된 사양" | "인식된 양식 정보" |
| 필드 카드 | 이름·타입·형식·필수 배지 | 좌동 + 신뢰도 배지("확인됨" / "확인 필요") |
| 진행률 표시 | coverage 기반 근사치(F1 §8.2) | 좌동(§4.3 루브릭 대비 채워진 항목 비율) |
| 하단 버튼 | "생성"(spec_complete 시 활성) | "이대로 등록하기"(spec_complete 시 활성) |
| 결과 영역 | 새로 저작된 워크북 미리보기 + 대화형 수정 입력창(F1-7) | `structural_json` 기반 원본 구조 미리보기(read-only 표) + field_rules 검수 표(직접 수정 가능 → §5.4 PATCH) + **대화형 수정 입력창 비활성**(툴팁 "등록된 원본 파일은 수정할 수 없습니다. 작성 기준만 수정할 수 있습니다") |
| 첨부 처리 | 참고 자료로만 사용 | 첫 xlsx 첨부 시 자동으로 `primary_attachment_id` 후보 지정. xlsx 2개 이상 첨부 시 라디오 버튼으로 "등록 대상" 1개를 명시적으로 선택하는 UI 노출(§5.2). hwpx는 "참고 자료로만 활용되며 등록 대상이 될 수 없습니다" 인라인 안내, png는 첨부 자체 거부 |
| 의도 모호 시 | 해당 없음 | 리플라이 버블 아래 "새 양식 만들기" / "첨부한 파일 그대로 등록" 버튼 2개 인라인 노출(§4.2) |

- 모드 전환 시점: 첫 응답의 `mode` 필드로 화면 톤이 전환되며, 챗봇 리플라이 버블 UI 자체는 F1과 동일한 컴포넌트를 재사용한다.
- TanStack Query 키는 F1과 동일(`['intake-session', projectId]`), `register` 호출은 동기이므로 폴링(`refetchInterval`)이 필요 없다 — mutation 성공 시 즉시 `['form-template', templateId]` 캐시를 채운다.

## 9. 보안·운영

### 9.1 AI 전송 동의

F1 §9.1 동의 문구를 재사용하되 recognize 모드 특이사항을 추가 고지한다: "업로드하신 양식(예시 데이터가 포함될 수 있음)의 구조와 샘플 값 일부가 OpenAI API로 전송됩니다."

### 9.2 개인정보 마스킹 — 리스크 강화 지점

- F1 §9.2 마스킹 파이프라인을 `structural_json.sample_rows`에도 동일 적용(주민번호·전화번호·이메일 정규식 치환).
- **F6 고유 리스크**: 예시 데이터 파일은 실제 회신 데이터일 가능성이 F1의 일반 참고 첨부보다 높다(§10 R5). 이에 따라 recognize 모드에서는 성명 열 마스킹 기본값을 F1(기본 미마스킹)과 달리 **"마스킹함"으로 전환**한다(§12 J4).

### 9.3 원본 파일 무결성

- §7 Registrar의 매직바이트 검증 + 매크로 포함 확장자(.xlsm 등) 거부.
- 바이러스 스캔 연동은 MVP 범위 밖(자리표시자, v2 검토, §12 J5).

### 9.4 감사 로그

`audit_logs`에 F1 액션에 더해 다음을 기록: `form_recognize_message`, `form_register`, `form_field_rules_update`, `form_reregister`(actor_id, target=template/session id).
